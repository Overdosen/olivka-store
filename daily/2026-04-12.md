# Daily Log: 2026-04-12

## Sessions

### Session (10:00 - 19:30) - Оптимізація Чекауту та Платіжних Потоків

**Context:** Користувач хотів фіналізувати процес оплати, забезпечити миттєве оновлення статусу замовлення в інтерфейсі та автоматизувати списання залишків товару.

**Key Exchanges:**
- Обговорили логіку LiqPay: вирішили не змінювати існуючу логіку `paid`, а лише додати автоматичне оновлення сторінки успіху.
- Виявили проблему зі списанням товару при ручній зміні статусу: вирішили перенести логіку в базу даних (тригери).
- Користувач надав конкретні вимоги до дизайну сторінки Післяплати (Screenshot #2) — мінімальне заокруглення, чіткі реквізити, IBAN блок.

**Decisions Made:**
- **Supabase Realtime**: Використовувати підписку на `postgres_changes` замість інтервального опитування (`polling`).
- **Database Triggers**: Реалізувати списання залишків на рівні PostgreSQL, щоб гарантувати цілісність даних незалежно від джерела оновлення (Webhook або Admin Panel).
- **Replica Identity**: Встановити `REPLICA IDENTITY FULL` для таблиці `orders`, щоб Realtime отримував повні дані рядка.

**Lessons Learned:**
- Supabase Realtime при налаштуваннях за замовчуванням може не надсилати всі колонки — завжди перевіряй `REPLICA IDENTITY`.
- Для критичних станів (оплата) краще мати подвійну перевірку: Realtime + запасне фонове опитування (fallback).
